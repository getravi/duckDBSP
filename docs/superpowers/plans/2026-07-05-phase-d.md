# Phase D: Vectorized Evaluation, Outer Joins, Subqueries — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans.
> Same conventions as Phase C (docs/superpowers/plans/2026-07-05-phase-c-planner-completion.md):
> TDD via the differential harness in test/integration/test_planner_frontend.cpp,
> build/test in test/build_test, full suite green + docs per task, one commit per task.

**Goal:** Close the three evidence-ranked gaps from pre-Phase-D: expression-eval
throughput (4.6× benchmarked overhead), outer joins, and the subquery shapes
that don't need DELIM_JOIN.

**Baseline:** 31/31 tests green; bench_planner_eval: fused filter 259k rows/s,
hand lambda 1.2M rows/s, aggregate 770k rows/s, join delta 140k rows/s.

## Verified plan shapes (probed 2026-07-05, optimizer off)

```
LEFT/RIGHT/FULL:      COMPARISON_JOIN with join_type field (same operator as inner)
IN (subquery):        FILTER(SUBQUERY-ref) → COMPARISON_JOIN Join Type: MARK → children
scalar subquery cmp:  FILTER(val > SUBQUERY) → CROSS_PRODUCT(t, PROJECTION(CASE err)
                        → AGGREGATE[first(#0), count_star()] → PROJECTION(avg) → AGGREGATE)
EXISTS / correlated:  DELIM_JOIN (Join Type: MARK) + DELIM_GET — stays E110
```

Key facts: MARK join output = left columns + one BOOLEAN "mark" column the FILTER
above consumes; NOT IN needs null-aware marks (right side containing a NULL key
makes an unmatched mark NULL, not false). Scalar subqueries only need the
`first` aggregate — CASE/error()/CROSS_PRODUCT/AGGREGATE already translate.

### Task D1: Vectorized expression evaluation (filter/map/filter_map)

**Files:** `include/dbsp_plan_translator.hpp`; bench comparison via
`test/benchmarks/bench_planner_eval.cpp` (no changes needed — rerun).

**Design:** `ChunkedEval` next to `RowExprEval`: owns one ExpressionExecutor and
input DataChunk sized STANDARD_VECTOR_SIZE; `eval(rows) → duckdb::Vector`.
Nodes buffer delta entries `(const DuckDBRow*, weight)` into batches of 2048,
evaluate each expression once per batch, then assemble output rows from result
vectors with `Vector::GetValue(i)`. Applies to the build cases for FILTER_EXPR,
MAP_EXPR, and FILTER_MAP (replace their per-row RowExprEval lambdas with
batch-processing node classes). RowExprEval stays for join/aggregate/window
(per-row by structure; revisit only if D1 numbers justify).

Steps: correctness first (suite must stay green — differential tests already
cover these paths heavily incl. NULLs), then rerun bench_planner_eval and
record before/after in the commit message. Target: ≥3× on the fused filter
path. If a batch path can't match row-path semantics (e.g. error-throwing
expressions must not fire for filtered-out rows — check CASE/error() from D3
shapes: DuckDB evaluates eagerly over the chunk, same as its own execution,
so this matches engine semantics), document any deviation found.

### Task D2: Outer joins (LEFT / RIGHT / FULL)

**Files:** `include/dbsp_plan_translator.hpp` (`visit_join`, `PlanJoinNode`);
tests in test_planner_frontend.cpp.

**Design:** PlanJoinNode learns `join_type`. Keep the bilinear inner core.
Add per-side NULL-padding state:
- `left_pad_`: for each distinct left row, its weight and weighted match count
  (equi-key + residual-passing pairs). When match count crosses 0 (either
  direction) or the row's weight changes while unmatched, emit/retract
  `(left ⧺ NULLs)` with the row's weight.
- RIGHT mirrors with `(NULLs ⧺ right)`; FULL maintains both.
Match-count maintenance rides the existing delta passes: Δr updates counts of
matching left rows; Δl computes the initial count from the right index.
Output schema unchanged (left cols + right cols); NULL padding uses typed NULL
`duckdb::Value(type)` per column.

Tests: differential LEFT/RIGHT/FULL over randomized two-table rounds
(runDifferentialTwoTables — NULL join keys included), plus a deterministic
transition test: unmatched row appears padded → gains its first match (pad
retracted) → loses it (pad back) → left row deleted while unmatched.

### Task D3: MARK joins + FIRST aggregate → IN / NOT IN / scalar subqueries

**Files:** `include/dbsp_plan_translator.hpp`; tests in test_planner_frontend.cpp.

**Design:**
1. `PlanAggSpec::Fn::FIRST` — weight-tracked multiset like MIN, emits
   `*begin()`; in the shapes that produce it the input is a single row, so any
   deterministic choice is exact.
2. MARK join in PlanJoinNode: output = left row ⧺ one BOOLEAN mark column.
   State per distinct left row: weighted match count + count of right rows
   with NULL join key (null-aware NOT IN: mark = true if matches > 0, NULL if
   no match but right NULLs exist, else false). Transitions retract/emit the
   marked row. The FILTER above consumes the mark as an ordinary boolean
   column (bound-ref) — no special casing there.
3. visit_join: accept JoinType::MARK (columns = left + {"mark", BOOLEAN}) and
   LEFT/RIGHT/FULL from D2; SEMI/ANTI arrive only from optimizer rewrites we
   don't run — reject with E110 naming them if ever seen.

Tests: IN, NOT IN (incl. a NULL in the subquery side — differential harness
catches the three-valued semantics), `val > (SELECT AVG(...))` end-to-end incl.
incremental updates to the inner table; error_handling_e2e's subquery test
gets updated intent if the uncorrelated case now succeeds (it uses a
correlated-equivalent same-table shape — verify which plan it produces).

### Task D4: Closeout

Docs (ARCHITECTURE operator list + diagram row, README supported/unsupported
lists, errors E110 page, CHANGELOG), memory update, TODO.md limitations list

## Risks

- D1 touches the hottest correctness path — the differential suite is the
  gate; land it before D2/D3 so their tests also run on the batched path.
- Outer-join weight bookkeeping (multiplicities > 1, weight changes while
  unmatched) is the fiddly part — deterministic transition test pins it.
- NOT IN null-awareness is easy to get subtly wrong — differential test with
  NULLs on both sides is mandatory.
