duckDBSP · a DuckDB extension, explained without the jargon

Answers that update themselves.

duckDBSP keeps live answers — running totals, filtered lists, joined reports — inside the DuckDB database. When your data changes, it doesn't recompute the answer from scratch. It reads only the change, and nudges the answer.

The problem

Recounting the warehouse vs. reading the receipt

Say you run a shop and keep a board on the wall: “Total sales per customer.” A new order arrives. How do you update the board?

The usual way

Walk through every order ever placed, add them all up again, rewrite the board. Fine for ten orders. Painful for ten million — and you redo it after every single change.

Work grows with your history

The duckDBSP way

Alice, $100

Look at just the new receipt. Alice's line on the board goes up by $100. Done. Nobody re-reads the old orders — they were already counted.

Work grows only with the change
The trick

Every change is a ledger entry with a sign

Inside duckDBSP, data never moves around as big tables. It moves as tiny ledger entries: a row, plus a sign. means “this row arrived.” means “this row left.” An edit is simply both at once — the old row leaves, the new one arrives.

That's the whole secret. Deletes aren't special, edits aren't special; everything is arrivals and departures flowing through the same machine. (Researchers call this a Z-set; you can call it a signed ledger.)

It stacks

Answers built on answers

Live answers can feed other live answers. When the orders table changes, the dispatcher delivers changes downhill in dependency order — each level nudges the next, and nothing is ever recomputed from scratch.

Good to know

The fine print, kept short

Reads don't wait

Many people can read the live answers at the same time. Only changes (new data, new views) take a brief exclusive turn.

It lives in memory

Live answers are kept in RAM for speed. The definitions are saved; after a restart, answers are rebuilt from your tables.

It can follow chains

“Who's connected to whom, through any number of hops?” Recursive questions are answered by looping until nothing new appears.

It tidies the line first

Before building an assembly line, an optimizer rearranges it — filtering early, dropping unused columns — so less data rides through.
