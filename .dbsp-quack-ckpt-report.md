## Files Changed

### `include/dbsp_cdc.hpp`

**1. Added `qualify()` helper (after `is_valid_identifier`)**

```cpp
inline std::string qualify(const std::string &cat, const std::string &tbl) {
  return cat.empty() ? ("\"" + tbl + "\"") : ("\"" + cat + "\".\"" + tbl + "\"");
}
```

Produces `"m"."_dbsp_views"` when catalog is non-empty, `"_dbsp_views"` otherwise.

**2. `save_checkpoint(context, catalog="")` — new `catalog` parameter**

- Validates `catalog` with `is_valid_identifier` (rejects dotted/injected input).
- Qualifies `_dbsp_ckpt` and `_dbsp_ckpt_meta` with `qualify(catalog, ...)`.
- All `CREATE OR REPLACE TABLE` and `INSERT INTO` references use the qualified name.

**3. `checkpoint_valid(context, out, catalog="")` — new `catalog` parameter**

- Existence check scoped to the right catalog:
  - Default: `WHERE table_name = '_dbsp_ckpt'`
  - With catalog: `WHERE table_catalog = 'catalog' AND table_name = '_dbsp_ckpt'`
- Reads from `qualify(catalog, "_dbsp_ckpt")` and `qualify(catalog, "_dbsp_ckpt_meta")`.

**4. `save_to_duck_table(context, storage_table="_dbsp_views", only_view="", catalog="")` — new `catalog` parameter**

- Validates `catalog` with `is_valid_identifier`.
- Computes `qtable = qualify(catalog, storage_table)` once.
- All SQL references use `qtable`.
- **Critical bug fixed here**: the `prep->Execute(def.name, def.sql, ...)` call was passing `std::string` arguments through `Value::CreateValue<string>()`, which creates a **BLOB** value (not VARCHAR). When a BLOB is stored into a VARCHAR column, DuckDB hex-encodes the bytes — double-quote characters become `\x22`, making the stored SQL unparseable on reload. Fixed by explicitly constructing `duckdb::Value(string)` objects (which create VARCHAR) and calling `Execute(vector<Value>&)`.

**5. `load_from_duck_table(context, storage_table="_dbsp_views", catalog="")` — new `catalog` parameter**

- Validates `catalog` with `is_valid_identifier`.
- Computes `qtable = qualify(catalog, storage_table)`.
- Existence check scoped to catalog (same pattern as `checkpoint_valid`).
- Reads from `qtable`.
- **Critical bug fixed here**: row reading was using `res->GetValue(i).ToString()` which for VARCHAR values escapes characters like `"` as `\x22`. Fixed by using `duckdb::StringValue::Get(val)` which returns the raw string content.
- Passes `catalog` through to `checkpoint_valid(context, ckpt, catalog)`.

## Qualify Approach

`qualify(cat, tbl)` is a minimal helper that always double-quotes both parts when a catalog is given. Since `is_valid_identifier` rejects dots and special characters, both `cat` and `tbl` are guaranteed to be safe plain identifiers before quoting. No injection is possible.

---

## Load-Side Existence Check Scoping

DuckDB's `information_schema.tables` includes a `table_catalog` column that holds the attached database name. When `catalog="m"`, the existence check becomes:

```sql
SELECT 1 FROM information_schema.tables
WHERE table_catalog = 'm' AND table_name = '_dbsp_views'
```

This correctly scopes the lookup to the attached file DB rather than the default in-memory catalog.

## Attach-Mode Load Outcome

Works correctly. The `InternalQueryGuard` + fresh `duckdb::Connection(DatabaseInstance::GetDatabase(context))` pattern already sees attached catalogs, so `CREATE TABLE "m"."_dbsp_views"` and reads from it work transparently.

The view SQL stored in `_dbsp_views` already contains qualified refs (`"m".li_0`). When `create_view` reconstructs a view from that SQL, `PlanTranslator::translate` plans it against the same database instance (which has `m` attached) and the source tables resolve correctly.

### Bugs discovered and fixed

Two pre-existing bugs were found in the round-trip path:

1. **Save bug** (`PreparedStatement::Execute` with `std::string`): `Value::CreateValue<string>` creates a BLOB, not VARCHAR. BLOBs stored in VARCHAR columns are hex-encoded by DuckDB, corrupting double-quoted identifiers in SQL strings. Fixed by explicitly building a `duckdb::vector<duckdb::Value>` with `duckdb::Value(string)` constructors (VARCHAR) and calling `Execute(vector<Value>&)`.

2. **Load bug** (`Value::ToString()` on VARCHAR results): `duckdb::Value::ToString()` escapes `"` as `\x22` in its output. Fixed by using `duckdb::StringValue::Get(val)` to extract raw VARCHAR content without escaping.

Both bugs were latent in the default-catalog path too, but only manifested with double-quoted identifiers in view SQL (the existing `test_checkpoint_restore.py` test doesn't use quoted identifiers). The fixes benefit both paths.
